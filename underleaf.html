01 — Vision

The Interactive Document Engine

LaTeX tooling has not meaningfully advanced in a decade. TeXstudio compiles documents. Overleaf hosts them. Neither treats the document as a living artifact — one that reacts to data, explains its own errors, and previews itself as you write. Textura does.

The core thesis: a LaTeX document is the frontend for your research. It should behave like one — reactive, fast, and integrated with the data sources that feed it. This is achievable today with the right tooling: Tectonic eliminates setup friction, Ollama brings local AI without a server, and Go's concurrency model makes real-time file watching trivially fast.

04 — Novel Features

What Neither Overleaf nor TeXstudio Does

These features are the product's reason to exist. Each integrates an existing tool in a way that has not been done in a LaTeX editor before. The engineering is integration work — the novelty is the combination.

Feature A — Data Integration

Live CSV / JSON Binding

Link a data file to a LaTeX table or figure. When your model output changes, fsnotify detects it and the document updates automatically. LaTeX becomes the view layer for your data.

06 — Architecture

Data Flow

The compilation loop follows a five-step cycle driven by a 500ms debounce on keystroke events:

1. File Change → user stops typing (500ms debounce)
2. Incremental Parse → Go backend updates AST for changed file only
3. Tectonic Call → backend invokes engine; missing packages fetched to local cache
4. Signal → Wails emits compile:success event to frontend
5. Refresh → PDF.js reloads from blob URL; Ghost Previews update from AST delta

For error paths, the same loop emits compile:error with the raw log. The Ollama integration runs on a separate Go routine — log translation never blocks the UI.

08 — Roadmap

Phase 1 Implementation Order

Ordered by dependency and proof-of-concept value. Each phase produces a working, demonstrable artifact.

Phase 1.1 · Skeleton

Wails Project + Split-Pane Layout

Initialize the Wails v3 project. Wire up a basic Svelte split-pane with a CodeMirror instance on the left and a placeholder PDF viewer on the right.

Phase 1.2 · Engine Bridge

Tectonic Compile Loop

Implement the Go wrapper for Tectonic. Compile a "Hello World" .tex file and display the result in PDF.js. Establish the compile:success / compile:error event bus.

Phase 1.3 · AST Layer

Go Participle Parser

Build the grammar for \section, \label, math environments, and the @bind tag. This AST powers Ghost Previews and Data Binding. It's the platform everything novel is built on.

Phase 1.4 · Ghost Previews

Inline KaTeX via CodeMirror Decorations

Use the CodeMirror 6 decoration API to overlay KaTeX-rendered math above the line being edited. Pull the math source from the AST, render it, inject the widget decoration.

Phase 1.5 · Error AI

Ollama Log-Physician Panel

On compile:error, ship the raw TeX log to a local Ollama instance on a separate Go routine. Stream the response into a dedicated error panel. Add one-click fix application.

Phase 1.6 · Data Binding

Live CSV/JSON Watcher

Use fsnotify to watch bound data files. On change, re-generate the linked LaTeX table and trigger a recompile. This is the feature that defines the product's identity.
